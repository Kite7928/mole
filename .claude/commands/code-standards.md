# 代码质量标准速查

快速查阅项目强制代码标准，编码时随时参考。

---

## 📝 语言规范

| 场景 | 语言 | 示例 |
|------|------|------|
| 代码注释 | 简体中文 | `# 获取用户列表` |
| 文档 | 简体中文 | README、设计文档 |
| Git提交 | 简体中文 | `feat: 添加用户评论功能` |
| 变量/函数名 | 英文 | `get_user_list()` |
| 类名 | 英文 | `ArticleService` |

编码：所有文件 UTF-8 无 BOM。

---

## 🏗️ 设计原则

| 原则 | 要求 |
|------|------|
| **S** 单一职责 | 每个函数/类只做一件事 |
| **O** 开闭原则 | 对扩展开放，对修改关闭 |
| **L** 里氏替换 | 子类可替换父类 |
| **I** 接口隔离 | 接口专一，不要胖接口 |
| **D** 依赖倒置 | 依赖抽象而非具体实现 |
| **DRY** | 重复3次以上才抽象，不过早抽象 |

---

## ❌ 禁止事项

- MVP、最小实现、占位符
- 自研方案（必须优先复用官方SDK/社区方案）
- "聪明"技巧（可读性优先）
- 过早抽象和过度架构
- "修改说明"式注释（变更信息交给git）
- 未经评估的昂贵依赖或阻塞操作

---

## ✅ 必须事项

- 完整实现全量功能与数据路径
- 遵循项目既有代码风格（导入顺序、命名、格式化）
- 评估时间复杂度、内存占用与I/O影响
- 复杂逻辑先拆分职责再编码
- 主动删除过时、重复代码
- 破坏性改动提供迁移步骤

---

## 🧪 测试要求

| 类型 | 覆盖范围 |
|------|---------|
| 正常流程 | 主要业务路径 |
| 边界条件 | 空值、极值、并发 |
| 错误恢复 | 异常处理、降级策略 |

缺失测试 = 风险，必须记录补测计划。

---

## ⚡ 性能检查

- [ ] 时间复杂度是否合理？
- [ ] 内存占用是否可控？
- [ ] I/O操作是否必要？
- [ ] 是否有阻塞操作？
- [ ] 外部依赖是否经过评估？

---

## 📊 架构优先级

```
官方SDK/社区方案 > 项目内复用 > 新建实现（需特例批准）
```

自研实现必须规划替换或下线时间表。
